# Arquitectura Inicial, Persistencia y Estructuras de Datos

## Estado

Aceptado.

## Contexto

El proyecto tiene como objetivo implementar un sistema de logística para un centro de distribución, cumpliendo los requerimientos de localización de stock, optimización de recorridos, control de inventario crítico, expedición de pedidos y trazabilidad de mercadería.

La consigna evalúa principalmente el uso de estructuras de datos y algoritmos, por lo que se decidió priorizar una arquitectura simple que permita evidenciar claramente la utilización de los TDA seleccionados.

---

## Decisiones Adoptadas

### Persistencia

Se decidió utilizar archivos JSON como mecanismo de persistencia inicial del sistema.

Los archivos JSON almacenarán los datos de productos, lotes, movimientos, pedidos, usuarios y demás entidades necesarias.

No se utilizará una base de datos relacional o NoSQL durante la primera etapa del proyecto.

---

### Carga de Datos

Al iniciar la aplicación, se ejecutará un proceso de carga inicial mediante la clase DataLoader.

Su responsabilidad será:

* Leer los archivos JSON.
* Convertir los registros a objetos Java.
* Construir las estructuras de datos necesarias.
* Cargar dichas estructuras en memoria.

Una vez finalizada la carga, la aplicación trabajará exclusivamente sobre las estructuras almacenadas en memoria.

---

### DataStore Central

Se decidió implementar una clase DataStore como almacenamiento central de las estructuras de datos activas del sistema.

El DataStore funcionará como punto único de acceso para los servicios de aplicación.

Las estructuras mantenidas serán:

```text
AVLTree<Producto> productos

AVLTree<Lote> lotes

MinHeap<Producto> inventarioCritico

DynamicQueue<Pedido> expedicion

IDynamicStack<Movimiento> movimientos

Graph deposito
```

---

### Separación entre Persistencia y Lógica

Los archivos JSON serán utilizados únicamente como persistencia.

Las operaciones de búsqueda, inserción, eliminación y actualización serán realizadas sobre las estructuras cargadas en memoria.

De esta forma se evita realizar consultas directas sobre archivos y se aprovechan las complejidades temporales de los TDA implementados.

---

### Modelado de Productos y Lotes

Se decidió separar los conceptos de Producto y Lote.

La relación será:

```text
Producto
    |
    | 1:N
    |
Lote
    |
    | 1:N
    |
Movimiento
```

Los movimientos de mercadería estarán asociados a lotes y no directamente a productos.

Esta decisión permite implementar trazabilidad completa y registrar el recorrido de cada partida de mercadería dentro del sistema.

---

### Localización de Stock

Para localizar productos y lotes sin realizar recorridos lineales se utilizarán árboles AVL.

Las búsquedas se realizarán sobre:

```text
AVLTree<Producto>

AVLTree<Lote>
```

La complejidad esperada será O(log n).

---

### Inventario Crítico

Para identificar rápidamente los productos con menor existencia se utilizará un Min Heap.

La estructura permitirá obtener de forma eficiente los productos que requieren reposición prioritaria.

---

### Expedición de Pedidos

La línea de expedición será modelada mediante una cola FIFO.

Los pedidos preparados ingresarán al final de la cola y serán despachados respetando el orden de llegada.

---

### Trazabilidad y Reversión de Operaciones

Los movimientos recientes serán almacenados en una pila.

La estructura permitirá implementar operaciones de reversión siguiendo el principio LIFO.

Cada nuevo movimiento será agregado mediante push y las operaciones de corrección podrán recuperar el último movimiento mediante pop.

---

### Optimización de Recorridos

La distribución física del depósito será representada mediante un grafo.

Los pasillos actuarán como nodos y las conexiones entre ellos como aristas.

Las rutas mínimas podrán calcularse utilizando el algoritmo de Dijkstra.

---

### Arquitectura de Aplicación

La aplicación se organizará mediante una arquitectura por capas simplificada:

```text
Controllers
    ↓
Services
    ↓
DataStore
    ↓
Estructuras de Datos
    ↓
Persistencia JSON
```

Los Controllers expondrán la API.

Los Services contendrán la lógica de negocio.

El DataStore mantendrá las estructuras activas.

Los TDA implementarán las operaciones optimizadas requeridas por la consigna.

Los archivos JSON actuarán como persistencia del sistema.

---

## Consecuencias

### Positivas

* Arquitectura simple y fácil de comprender.
* Uso explícito de los TDA requeridos por la consigna.
* Separación clara entre persistencia y lógica de negocio.
* Búsquedas y operaciones optimizadas mediante estructuras especializadas.
* Facilidad para justificar decisiones durante la defensa del proyecto.

### Negativas

* Mayor consumo de memoria debido a la carga completa de datos.
* Persistencia basada en archivos, con menor escalabilidad que una base de datos.
* Necesidad de sincronizar memoria y archivos cuando se produzcan modificaciones.
