# ADR 001 — Arquitectura Cliente-Servidor Contenerizada

## Estado

Aceptado.

---

# Contexto

El sistema “Centro Logístico de Distribución Avanzada” requiere separar responsabilidades entre interfaz de usuario y lógica de negocio para facilitar organización del proyecto, mantenimiento y despliegue.

Además, se busca mantener aislamiento entre servicios internos y reducir exposición innecesaria de componentes críticos del sistema.

---

# Decisión

Se adoptó una arquitectura cliente-servidor contenerizada utilizando Docker y Docker Compose.

La solución se compone de:

* Frontend web servido mediante Nginx.
* Backend API desarrollado con Spring Boot.
* Red interna de Docker para comunicación entre contenedores.

---

# Orquestación

Docker Compose se utiliza para gestionar y ejecutar los contenedores del sistema de forma centralizada.

Los servicios principales son:

* frontend
* backend

Ambos contenedores se comunican mediante una red interna compartida.

---

# Proxy Inverso

Nginx funciona como único punto de acceso público del sistema utilizando el puerto:

```text id="xmhf1d"
3000
```

Las solicitudes dirigidas a:

```text id="tq08ne"
/api/
```

son redireccionadas automáticamente hacia el backend interno.

---

# Aislamiento del Backend

El backend Spring Boot utiliza el puerto interno:

```text id="a8jv4m"
8080
```

Este puerto se encuentra expuesto únicamente dentro de la red interna de Docker mediante:

```text id="dlk1uo"
expose
```

No existe acceso externo directo al backend desde fuera de los contenedores.

---

# Organización Modular

El backend se organiza utilizando separación de responsabilidades en módulos y capas:

```text id="mjlwmv"
controllers/
services/ acciones requeridas por los usuarios 
models/ clases importantes tendran interfaz las mas complejas 
db/ (archivos json que funcionan como base de datos)
interfaces/
tda's implementados en clase a mano 
main sirve el servidor  
podria haber un main especifico por accion o feature para testear en terminal 


El frontend se organiza utilizando:

views/
services/
config/ 
```

---

# Consecuencias

## Positivas

* Separación clara entre frontend y backend.
* Mayor organización modular.
* Menor exposición del backend.
* Centralización del acceso mediante Nginx.
* Facilidad de despliegue utilizando Docker Compose.
* Mayor mantenibilidad del proyecto.

---

## Negativas

* Mayor complejidad de configuración inicial.
* Dependencia de sincronización entre rutas frontend, proxy y backend.
* Necesidad de reconstrucción de contenedores ante cambios estructurales.

---

# Decisiones Relacionadas

* Uso del prefijo global:

```text id="k6xrx6"
/api
```

en controladores Spring Boot.

* Centralización de rutas frontend mediante:

```text id="n3g4tt"
config.js
```

* Separación entre archivos estáticos y endpoints de API.
